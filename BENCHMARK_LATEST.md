## 5. 瓶颈分析与优化建议

### 5.1 已确认瓶颈

1. **全局写锁竞争**：`RwLock::write()` 在 SET 高并发时成为单点瓶颈，QPS 无法突破 50k（SET）
2. **P99 长尾延迟**：来自锁等待、Tokio 调度、首次 `write()` 触发惰性删除
3. **超大规模并发**：10,000+ 连接时 TCP 连接管理成为瓶颈，大量 `Broken pipe`

### 5.2 推荐优化路线（按优先级）

| 优化项 | 工作量 | 预期效果 | 优先级 |
|--------|--------|---------|------|
| 替换 `RwLock<HashMap>` 为 `DashMap` | 半天 | SET QPS +30~50%，P99 降 50% | **首推** |
| 按 key hash 分片到多 `RwLock<HashMap>` | 1 天 | 类似 DashMap，但需自行实现 | 备选 |
| TTL 清理独立线程 | 2 小时 | P99 降低 1~2ms | 简单有效 |
| 用 `bytes::Bytes` 零拷贝替代 `String` | 1 天 | P99 降低 1ms | 精细优化 |
| 支持 `MSET` 批量写入 | 半天 | 写入吞吐量翻倍 | 场景优化 |

### 5.3 与 Redis 的对比

## 6. 结论

作为"一个月可完成的 Rust 全栈学习项目"，Mini-Cache v0.3.0 在 WSL2 环境下：

- **功能层面**：完全达标（编译、测试、协议、API 监控面板均正常）
- **性能层面**：GET 读性能远超目标，SET 写性能接近目标，P99 延迟有优化空间
- **扩展层面**：单实例 1,000 并发稳定，超过 2,000 需考虑连接池或架构优化

后续按 `DashMap` 替换 → 分片存储 → 独立 TTL 线程的路线迭代，可预期将 QPS 和 P99 同时推升到生产可用级别。
